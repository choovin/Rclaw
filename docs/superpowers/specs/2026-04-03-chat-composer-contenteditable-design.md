# Chat 轻量 Composer（contenteditable）与内联 chip + `/`、`@` 下拉

## 文档关系

- **取代范围**：本设计取代 [`2026-04-03-chat-skill-picker-design.md`](./2026-04-03-chat-skill-picker-design.md) 中关于 **「透明 textarea + 镜像 overlay」** 作为输入主表面的方案；该文档中 **技能按钮、`SkillPickerPopover`、命令规范化、token 边界规则、i18n 与测试思路** 仍可作为补充参考，除非与本篇冲突时以本篇为准。
- **动机**：镜像层与 `textarea` 在字体度量、折行、滚动同步上易出现「光标与所见文字不对齐」；后续还要在输入流内实现 **`/` 触发技能下拉**、**`@` 触发员工/Agent 下拉**，更适合 **单一编辑表面** 的轻量实现，而非继续堆叠镜像与坐标 hack。

## 目标与范围

### 目标

1. 聊天输入区使用 **受控 `contenteditable` composer** 作为唯一文字表面（不再使用透明 `textarea` + 上层镜像 div）。
2. **视觉**：已完成的 slash 技能 token 显示为 **可删除的 chip**（与现有产品预期一致）；后续 **`/`**、**`@`** 在输入过程中弹出 **下拉补全**（列表项与键盘导航在实现阶段细化）。
3. **数据**：发送给 Gateway 的仍是 **一条纯文本**（`onSend(text, ...)` 契约不变）；**不做**加粗/斜体/列表/链接等真正排版能力（**只要 chip + 下拉**）。

### 非目标

- 不引入 Lexical / Tiptap / ProseMirror 等完整文档模型（除非日后维护成本证明必须迁移）。
- 不将「可存盘 HTML」作为消息真相来源；若 DOM 内为展示需要存在少量 `span`，发送前必须 **序列化为纯文本**。
- 不保证 Gateway 一定执行某条 slash（与旧 spec 一致）。

## 架构概要

| 层级 | 职责 |
|------|------|
| Composer 宿主 | 单个 `contenteditable` 容器；承担输入、选区、多行高度与滚动。 |
| 纯文本真相 | 从 DOM **序列化**为字符串（推荐 `innerText` + 与旧实现一致的换行归一化）；所有发送、复制、历史记录均以此为准。 |
| Token 展示 | 对「完整 slash token」将对应区间渲染为 chip（`contenteditable="false"` 的内联节点或等价策略），删除按钮仅删除该 token 对应纯文本范围。 |
| 触发与下拉 | 监听输入与选区，在光标前子串上匹配 `/`、`@` 查询模式；用 `Range`/`getClientRects` 得到锚点，**浮层**展示列表（可选用 `@floating-ui/dom` 仅做定位）。 |
| 粘贴 | `paste` 默认 `preventDefault`，只插入 `text/plain`；可选使用 `contenteditable="plaintext-only"`（能力检测 + 降级）。 |
| IME | `compositionStart`/`compositionEnd` 期间避免结构性改写（插入/拆除 chip、批量替换），与旧 spec 精神一致。 |

## 与现有功能的关系

- **技能按钮 + `SkillPickerPopover`**：保留；插入仍使用规范化后的 `/${commandName} `，插入后更新 composer 内容与选区。
- **工具栏 `@` Agent 选择**：实现阶段二选一或并存——**(a)** 仅保留 toolbar 作为快捷方式；**(b)** 逐步以输入内 `@` 为主并收敛 UI。本篇不强制删按钮，但 **`@` 下拉的数据源与「路由到某 Agent」语义** 须与产品一致（见下文「`@` 数据源」）。
- **附件、拖拽、Enter 发送、Shift+Enter 换行**：行为保持不变；实现时验证焦点不落在 chip 内部导致无法输入等边界情况。

## Slash 命令与 chip（继承旧规则）

以下与 [`2026-04-03-chat-skill-picker-design.md`](./2026-04-03-chat-skill-picker-design.md) 对齐，除非技术实现改为「DOM 节点」时在删除语义上等价：

- **commandName 规范化**：`a-z0-9_`，最长 32（从 skill slug/id 转换）。
- **Token 识别边界**：完整 token 才渲染为 chip；前置/后置边界规则与旧 spec「识别规则」章节一致，避免误伤 URL/路径。
- **删除 chip**：删除 token 对应字符范围及可选后随单空格；选区与 caret 更新规则与旧 spec 一致。

## `/` 技能下拉（规划中）

- **触发**：在「可插入命令」的上下文中，用户输入 `/` 后进入查询态（具体正则与「是否必须在行首/词界」在实现时与 token 边界一致）。
- **行为**：展示 `enabled` 技能列表（可复用 store 过滤）；选中项插入 `/${commandName} ` 并关闭下拉；`Esc` 关闭；键盘上下与回车在实现阶段与无障碍要求一起验收。
- **与技能按钮面板**：共享数据源与规范化函数，避免两套逻辑漂移。

## `@` 员工下拉（规划中）

- **触发**：输入 `@` 后进入查询态，子串用于过滤列表。
- **数据源（待产品钉死，实现前必须确认）**：例如当前会话可 `@` 的 **Agent 列表**、**数字员工**目录、或后台员工接口；列表项展示名与插入到正文中的 **文本格式**（如 `@显示名 `、`@id `）必须固定并写入实现说明，且与「消息发送后由谁解析」一致。
- **与现有 `targetAgentId`**：若正文内 `@` 已能表达路由目标，可逐步让 toolbar chip 变为可选；本篇不强制一次改完路由模型。

## 错误处理与兼容

- **HTML 粘贴**：一律按纯文本处理。
- **降级**：若运行环境不支持 `plaintext-only`，依赖 `paste` + `beforeinput` 拦截保持纯文本。
- **可访问性**：composer 保留合适 `role`/`aria-*`；chip 删除与下拉项具备可命名操作。

## 测试策略（实现阶段）

- **单元**：`parseSlashTokens`（或继任模块）、纯文本序列化/反序列化、token 删除与边界用例（含重复 token、Windows 换行）。
- **E2E**：技能按钮插入 → chip 展示 → 删除 → 发送文本；规划中的 `/`、`@` 下拉各至少一条冒烟；多行滚动后下拉锚点不明显飘移。

## 实施顺序建议

1. 用 contenteditable composer 替换 textarea+镜像，保持发送字符串与旧行为一致（含 chip 展示与删除）。
2. 接入 `/` 下拉与 `@` 下拉（数据源确认后）。
3. 按需收敛 toolbar `@` 与路由状态机。
